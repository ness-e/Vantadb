# BM25 — Best Matching 25

##Definition

**BM25** (Best Matching 25) is a **relevance ranking** algorithm for full-text search. It is a probabilistic scoring function that classifies documents according to their relevance to a query, improving the classic TF-IDF model.

## Mathematical Formula

For a query $Q$ with terms $q_1, q_2, \ldots, q_n$, the BM25 score of a document $D$ is:

$$
\text{score}(D, Q) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\text{avgdl}}\right)}
$$

Donde:
- $f(q_i, D)$ = frecuencia del término $q_i$ en el documento $D$
- $|D|$ = longitud del documento (número de términos)
- $\text{avgdl}$ = longitud promedio de documentos en la colección
- $k_1 \in [1.2, 2.0]$ = saturación de frecuencia de término
- $b = 0.75$ = normalización por longitud de documento
- $\text{IDF}(q_i)$ = inverse document frequency

### Key Components

| Componente | Fórmula | Propósito |
|-----------|---------|-----------|
| **IDF** | $\log\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5}$ | Penalizar términos comunes |
| **TF** | $\frac{f \cdot (k_1 + 1)}{f + k_1 \cdot (1 - b + b \cdot \frac{|D|}{\text{avgdl}})}$ | Saturación de frecuencia |
| **Length Norm** | $1 - b + b \cdot \frac{|D|}{\text{avgdl}}$ | Penalizar documentos largos |

## How It Works

### Practical Example

**Corpus:** 1000 documents

**Query:** `"vector database"`

**Document A:**
- Contains: "base", "data", "vector"
- Length: 50 words
- BM25 Score: **12.4**

**Document B:**
- Contains: "base", "data" (but not "vector")
- Length: 200 words
- BM25 Score: **6.8**

**Document C:**
- Contains: "base" (10 times), "data" (5 times)
- Length: 30 words
- Score BM25: **8.2** (saturation per repetition)

### Final Ranking

1. **Document A** (12.4) — Most relevant
2. **Document C** (8.2) — Relevant but repetitive
3. **Document B** (6.8) — Less relevant

## Implementation in VantaDB

### Inverted Index

BM25 requires an **inverted index**:

```
Término → Lista de documentos que lo contienen

"base" → [doc1, doc3, doc7, doc12, ...]
"data" → [doc1, doc2, doc3, doc15, ...]
"vector" → [doc1, doc8, doc20, ...]
```

### Structure in VantaDB

```rust
pub struct Bm25Index {
    // Término → Posting list
    postings: HashMap<String, PostingList>,
    
    // Estadísticas globales
    total_docs: usize,
    avg_doc_length: f32,
    
    // Parámetros
    k1: f32,  // Default: 1.2
    b: f32,   // Default: 0.75
}

pub struct PostingList {

    term_frequencies: Vec<u32>,
    positions: Vec<Vec<u32>>, // For phrase search
}
```

## BM25 vs vector-search

| Dimensión | BM25 (Léxico) | [[hnsw]] (Vectorial) |
|-----------|---------------|---------------------|
| **Tipo de búsqueda** | Keyword matching | Similitud semántica |
| **Query** | Texto exacto | Vector embedding |
| **Captura** | Coincidencia de términos | Significado |
| **Ejemplo query** | "PostgreSQL" | "base de datos relacional" |
| **Fuerza** | Precisión en keywords | Comprensión semántica |
| **Debilidad** | No entiende sinónimos | Puede perder keywords exactos |

### Example of Complementarity

**Query:** `"How to configure PostgreSQL?"`

**BM25 finds:**
- Documents with "PostgreSQL" and "configure"
- ❌ Can't find documents about "configuring Postgres" (without "SQL")

**HNSW finds:**
- Documents on "configure database"
- ✅ Find "configure Postgres" (semantically similar)
- ❌ You may lose specific PostgreSQL documents

**Solution:** [[rrf]] merges both results.

## hybrid-search in VantaDB

VantaDB combines BM25 + HNSW using [[rrf]]:

```python
# busqueda-hibrida (default)
results = db.search(
    vector=embed("¿Cómo configurar PostgreSQL?"),
    text="configurar PostgreSQL",
    top_k=10,
    mode="hybrid"  # BM25 + HNSW + RRF
)
```

### Hybrid-search flow

```
Query: "configurar PostgreSQL"
    │
    ├─▶ BM25 Index ──▶ Lista 1: [doc5, doc12, doc23, ...]
    │
    ├─▶ HNSW Index ──▶ Lista 2: [doc3, doc7, doc12, doc45, ...]
    │
    └─▶ RRF Fusion ──▶ Ranking Unificado: [doc12, doc5, doc7, ...]
```

## BM25 parameters

| Parámetro | Rango | Default | Efecto |
|-----------|-------|---------|--------|
| **k1** | [1.2, 2.0] | 1.2 | Saturación de TF. Mayor = más peso a frecuencia |
| **b** | [0, 1] | 0.75 | Normalización de longitud. 0 = sin norma, 1 = norma completa |

### Cuándo Ajustar Parámetros

| Caso | k1 | b | Razón |
|------|----|----|-------|
| Documentos cortos (tweets) | 1.2 | 0.0 | Longitud uniforme |
| Documentos largos (papers) | 2.0 | 0.75 | Penalizar verbosidad |
| Keywords repetitivos | 1.0 | 0.75 | Menos saturación |
| Vocabulario técnico | 1.5 | 0.5 | Balance |

## Limitaciones de BM25

### 1. No Semantic Capture

```
Query: "carro rojo"
BM25 NO encuentra: "automóvil escarlata"
HNSW SÍ encuentra: (vectores similares)
```

### 2. Sensible a Tokenización

```
"don't" → ["don", "t"] vs ["don't"]
Resultados diferentes según tokenizador
```

### 3. Doesn't Handle Synonyms

```
Query: "laptop"
BM25 NO encuentra: "notebook", "portátil"
```

### 4. Requires Inverted Index

- **Memory:** Posting lists can be large
- **Build:** Slower than HNSW for updates
- **Updates:** Expensive reindexing
