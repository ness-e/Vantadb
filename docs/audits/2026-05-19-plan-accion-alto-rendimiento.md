# Plan de Acción de Alto Rendimiento - VantaDB (Fase 6 y Cierre de Gates)

Este documento representa el plan de acción consolidado, limpio de duplicidades y estructurado cronológicamente bajo un análisis de dependencias técnicas riguroso. Diseñado para transicionar VantaDB hacia su fase final de optimización y preparación para producción.

---

## Tabla de Tareas Priorizadas y Secuenciadas

| Nº | Nombre de la Tarea | Instrucciones Paso a Paso para su Resolución | Dificultad / Tiempo | Entregable Esperado |
| :--- | :--- | :--- | :--- | :--- |
| **1** | **Inyección de GIL en PyO3 (`py.allow_threads`)** | 1. Localizar los puntos de entrada expuestos en `vantadb-python/src/lib.rs` (`insert`, `search`, `delete`, `flush`, `rebuild_index`).<br>2. Envolver las llamadas pesadas que ejecutan código de Rust usando bloques `py.allow_threads(move \|\| { ... })`.<br>3. Asegurarse de que no se retengan referencias a objetos de PyO3 (`PyObject`, `&PyAny`) dentro del bloque liberado para evitar violaciones de acceso de memoria.<br>4. Ejecutar la suite de pruebas concurrentes de Python para garantizar que múltiples hilos operan sin deadlock y liberan el GIL. | **Bajo** <br>(1-2 días) | Módulo PyO3 con `py.allow_threads` en todos los puntos críticos de entrada y suite de pruebas de concurrencia de Python ejecutada exitosamente. |
| **2** | **Telemetría de Memoria Específica de OS (Bypass de `sysinfo`)** | 1. Identificar en `src/metrics.rs` o `src/telemetry.rs` el punto de recolección de uso de memoria (RSS/VSZ).<br>2. Implementar compilación condicional (`#[cfg(target_os = "linux")]`, `#[cfg(target_os = "macos")]`, `#[cfg(target_os = "windows")]`) para llamar a las APIs de bajo nivel correspondientes.<br>3. En Linux: Leer directamente `/proc/self/smaps_rollup` o `/proc/self/statm` para evitar parseos pesados.<br>4. En macOS: Usar `proc_pidinfo` con `PROC_PIDREGIONINFO`.<br>5. En Windows: Usar `QueryWorkingSetEx` a través de las bindings de `windows-sys` o `winapi`.<br>6. Envolver las llamadas en bloques de control de errores para silenciar excepciones de permisos (`EACCES`/`EPERM`) devolviendo `None` y un log de advertencia. | **Bajo** <br>(2-3 días) | Módulo de telemetría de memoria nativa integrado sin dependencias pesadas de `sysinfo`, validado en contenedores y entornos con sandbox restrictivos. |
| **3** | **Benchmark HNSW Puro en RAM (`benches/hnsw_pure.rs`)** | 1. Crear el archivo `benches/hnsw_pure.rs` utilizando la suite de benchmarking `criterion`.<br>2. Inicializar un grafo HNSW en memoria (RAM) pre-poblado con 10,000 vectores sintéticos (dimensión 1536).<br>3. Diseñar un caso de prueba para búsqueda (`graph.traverse()`) y otro para inserción de nodos (`graph.insert()`), aislando completamente la latencia de parsing de queries, filtros estructurados, E/S de disco y serialización JSON.<br>4. Guardar el baseline de ejecución con `cargo bench --bench hnsw_pure -- --save-baseline hnsw-pure-base`. | **Bajo** <br>(3 días) | Archivo `benches/hnsw_pure.rs` integrado en la suite de Cargo y baseline documentado para medir futuras optimizaciones del path crítico. |
| **4** | **Implementación de Distancia Euclidiana (`DistanceMetric::Euclidean`)** | 1. Añadir el enum `DistanceMetric::Euclidean` a las definiciones en `src/node.rs` y el config de índices.<br>2. Implementar la función de distancia euclidiana L2 en el cálculo de distancias del grafo HNSW.<br>3. Proveer soporte en el planificador (`src/planner.rs`) para propagar la métrica desde la query de usuario.<br>4. Crear pruebas unitarias verificando que el recall de Euclidiana es correcto sobre vectores sintéticos. | **Bajo** <br>(3-4 días) | Grafo HNSW con soporte nativo de distancia Euclidiana (L2) expuesto en la API y cubierto por pruebas unitarias. |
| **5** | **Ranking Explicable (`explain: true` en `VantaMemorySearchRequest`)** | 1. Añadir la bandera opcional `explain: bool` al struct de petición `VantaMemorySearchRequest` y su equivalente en el server MCP.<br>2. Modificar el motor de búsqueda en `src/sdk.rs` para capturar métricas intermedias durante la ejecución (puntuación de BM25 pura por término, peso de distancia de coseno/L2, y penalizaciones/bonificaciones de RRF).<br>3. Construir la estructura `VantaSearchExplanation` que consolide estos datos y devolverla cuando la bandera sea activa.<br>4. Validar que cuando `explain: false`, no haya degradación en la latencia de búsqueda (<1% de overhead). | **Medio** <br>(4-5 días) | API de búsqueda híbrida con soporte de explicabilidad detallada, estructurando el desglose matemático del scoring final de los hits devueltos. |
| **6** | **Optimización de SIMD para Distancias Vectoriales (AVX2 / NEON)** | 1. Localizar los bucles críticos de cálculo de distancias (Coseno y Euclidiana).<br>2. Añadir detección en tiempo de compilación y ejecución de características SIMD (`is_x86_feature_detected!("avx2")` o `is_aarch64_feature_detected!("neon")`).<br>3. Implementar versiones optimizadas usando intrínsecas del compilador o la biblioteca `simba` / `packed_simd` de forma segura.<br>4. Implementar el fallback a código escalar estándar si la CPU destino no soporta AVX2/NEON.<br>5. Medir el incremento de velocidad comparándolo con el baseline `hnsw-pure-base` generado en la Tarea 3. | **Medio** <br>(6-8 días) | Path crítico de cálculo de distancias vectoriales optimizado con SIMD (AVX2/NEON), mostrando mejoras de rendimiento de ≥2.5x en CPU compatible. |
| **7** | **Procesamiento Léxico (Stemming & Stopwords)** | 1. Crear la feature opcional `text-analysis` en `Cargo.toml`.<br>2. Integrar la caja `snowballstemmer` o `tantivy-tokenizer` para proveer tokenización inteligente.<br>3. Añadir soporte para cargar diccionarios de stopwords en formato lazy-loaded por idioma (inglés/español).<br>4. Modificar el indexador léxico y la generación de queries de texto para procesar los términos a través del pipeline de análisis antes del cálculo BM25. | **Medio** <br>(1 semana) | Pipeline de tokenización, stemming y stopwords integrado bajo feature flag, verificado mediante pruebas de recall léxico multilingüe. |
| **8** | **API de Backup & Restore (Exportación lógica y Cold Copy)** | 1. Diseñar el endpoint en `src/sdk.rs` para exportar lógicamente un namespace completo (`export_namespace`), serializando el grafo y los metadatos en un archivo consolidado.<br>2. Diseñar el endpoint inverso `import_namespace` para reconstruir la base de datos a partir de este snapshot lógico.<br>3. Implementar un mecanismo de respaldo físico en frío (cold copy) que ponga el storage en pausa temporal, forzando un flush a disco antes de copiar de forma segura los archivos físicos del backend.<br>4. Documentar los procedimientos de respaldo ante las diferencias del backend Fjall vs RocksDB. | **Medio** <br>(1 semana) | Interfaz de Backup/Restore integrada en el SDK y documentada operacionalmente en `docs/operations/BACKUP_POLICY.md`. |
| **9** | **OpenTelemetry + Logs Estructurados JSON** | 1. Integrar las cajas `tracing-opentelemetry` y `tracing-subscriber` con formato JSON.<br>2. Instrumentar spans explícitos en las fases operacionales críticas: replay de WAL, búsquedas HNSW/BM25, compactación física de disco y consolidación de caché.<br>3. Configurar el exportador para integrarse con Jaeger/Tempo de forma transparente.<br>4. Validar en un benchmark de estrés que el overhead de tracing es <3% bajo carga sostenida. | **Medio** <br>(1-2 semanas) | Módulo de observabilidad estructurada listo para producción (logs JSON parseables por Loki/Elasticsearch y spans distribuidos OpenTelemetry). |
| **10** | **Refactorización de `src/planner.rs` en Pipeline Desacoplado** | 1. Rediseñar `src/planner.rs` para eliminar dependencias mutables del motor de ejecución durante las fases de análisis lógico.<br>2. Definir las estructuras aisladas del pipeline: `Parser` (genera AST), `LogicalPlanner` (genera Plan Lógico optimizando RRF y filtros), y `PhysicalPlanner` (genera Plan Físico/IR ejecutable).<br>3. Implementar un *trait* ejecutor común en el motor que procese la IR física sin conocer el origen de la query. | **Alto** <br>(2 semanas) | Planificador de consultas completamente desacoplado y estructurado en capas aisladas (AST -> Plan Lógico -> Plan Físico). |
| **11** | **Migración a Estructuras Zero-Copy (Apache Arrow Layout)** | 1. Identificar las estructuras de datos intermedios que transportan vectores y metadatos en la frontera de ejecución.<br>2. Adaptar la representación intermedia de datos para usar layouts alineados en memoria compatibles con Apache Arrow IPC.<br>3. Reemplazar la serialización JSON interna en los puentes locales por lectura directa sobre los buffers de memoria mapeados, eliminando allocations redundantes. | **Alto** <br>(2 semanas) | Capa de serialización intermedia refactorizada a formato de memoria Arrow, reduciendo el consumo de CPU en la frontera de comunicación del servidor/SDK. |
| **12** | **Chaos Testing Expandido (Inyección de fallos en compactación, OOM y red)** | 1. Expandir `tests/storage/chaos_integrity.rs` incorporando inyección de fallos de E/S (`failpoints`) durante los ciclos activos de compactación en disco.<br>2. Diseñar un test que simule una caída por falta de memoria (OOM) matando el proceso de forma abrupta mientras se consolida la caché.<br>3. Incorporar simulaciones de pérdida de conexión y latencia de red en los hilos del servidor MCP.<br>4. Validar la auto-recuperación determinista al 100% sin corrupción silenciosa de datos. | **Alto** <br>(2 semanas) | Suite expandida de pruebas de caos automatizadas y validadas en el pipeline de CI pesado (`heavy_certification.yml`). |
| **13** | **Automatización de Publicación en PyPI (CI de Ruedas/Wheels)** | 1. Crear un workflow de GitHub Actions que compile nativamente los bindings de PyO3 (`vantadb_py`) para arquitecturas x86_64 y aarch64 sobre Linux, macOS y Windows usando `cibuildwheel` y `maturin`.<br>2. Configurar el firmado digital seguro de los paquetes resultantes empleando Sigstore.<br>3. Agregar validaciones automatizadas post-instalación en entornos virtuales de Python (venv) limpios. | **Medio** <br>(1 semana) | Workflow de CI tag-gated y firmado digitalmente listo para distribuir el SDK de Python en PyPI con un solo comando de release. |
| **14** | **Ejecución de Benchmark Competitivo contra Qdrant/LanceDB (SIFT1M)** | 1. Diseñar un pipeline de pruebas automatizado que monte VantaDB, Qdrant y LanceDB sobre el mismo hardware virtualizado de referencia (4 vCPUs, 8GB RAM).<br>2. Alimentar el benchmark con el dataset estándar SIFT1M (1 millón de vectores de 128 dimensiones).<br>3. Registrar métricas de throughput (consultas/seg), latencia P99, tasa de acierto (Recall@10) y consumo RSS de memoria.<br>4. Documentar los scripts de inicialización y publicar los resultados sin filtros de conveniencia. | **Alto** <br>(2 semanas) | Suite de benchmarks competitivos reproducible y reporte técnico final publicado en `docs/benchmarks/`. |
| **15** | **Despliegue y Validación en Pilotos Controlados** | 1. Seleccionar de 3 a 5 casos de uso internos o externos (ej. recuperación RAG local, memoria a largo plazo para agentes, caché semántica).<br>2. Desplegar VantaDB en dichos entornos monitoreando la estabilidad durante al menos 14 días continuos de uso real.<br>3. Establecer un SLA interno riguroso: latencia de búsqueda híbrida P99 < 10ms y cero pérdida de datos.<br>4. Recopilar feedback estructurado sobre fallos, facilidad de instalación y cuellos de botella para iteraciones de producto. | **Alto** <br>(3 semanas) | Informe técnico de pilotos cerrados con métricas operacionales reales e incidencias documentadas de fin de ciclo. |
