# P2-2: PyO3 — VantaVector.__array_interface__ UB

**Fuente:** Plan `code-freeze-action-plan.md`
**Esfuerzo:** 🟡 ~1 hr  
**Tipo:** Soundness fix  
**Archivos afectados:** `vantadb-python/src/lib.rs:1758-1824`

## Objetivo

`VantaVector.__array_interface__` exponía `self.data.as_ptr()` de un `Vec<f32>`. Si Rust reasignaba el Vec mientras NumPy referenciaba el buffer → UB (dangling pointer). Riesgo bajo porque `VantaVector.data` es privado y nunca se muta después de construcción, pero la superficie de UB existe.

## Cambio

`Vec<f32>` → `Box<[f32]>` en `VantaVector`:

- `Box<[f32]>` está heap-allocated con dirección estable (como Vec), pero no se puede reasignar (no tiene `push`/`reserve`)
- Elimina cualquier posibilidad de realloc mientras NumPy referencie el buffer
- `as_ptr()` retorna la misma dirección estable
- `__iter__` ahora usa `.to_vec()` en vez de `.clone()`
- `__getstate__`/`__setstate__` usan `.to_vec()`/`.into_boxed_slice()`

**Archivos modificados:** `vantadb-python/src/lib.rs:1758-1824` (~15 líneas cambiadas)

## Verificación

- `cargo check` en `vantadb-python/` ✅ (0 errores)
- No hay cambio en API pública (VantaVector sigue exponiendo `__array_interface__` con el mismo comportamiento)

## Ponytail note

`Vec<f32>` → `Box<[f32]>`. Elimina realloc como source de UB. Sigue habiendo UB si VantaVector se dropea mientras NumPy referencia — eso es inherente a `__array_interface__` y requeriría `Vec::leak()` (memory leak perpetuo). `Box<[f32]>` cubre el caso realista.
