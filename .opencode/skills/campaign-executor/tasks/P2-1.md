# P2-1: WASM — OpfsFile::delete() siempre devuelve error

**Fuente:** Plan `code-freeze-action-plan.md`
**Esfuerzo:** 🟢 ~15 min  
**Tipo:** Bugfix WASM  
**Archivos afectados:** `vantadb-wasm/src/opfs.rs:83-87`

## Objetivo

`OpfsFile::delete()` devolvía siempre `Err("OpfsFile::delete requires the parent directory handle — use OpfsStorage::delete_file instead")`. El método era `pub` pero inutilizable. El delete real estaba solo en `OpfsStorage::delete_file()`.

## Cambio

Reemplazar el stub de error con implementación real usando `FileSystemFileHandle.remove()` (Web API):

**Antes:**
```rust
pub async fn delete(&self) -> Result<bool, JsValue> {
    Err(JsValue::from_str(
        "OpfsFile::delete requires the parent directory handle ...",
    ))
}
```

**Después:**
```rust
pub async fn delete(&self) -> Result<bool, JsValue> {
    js_call(&self.handle, "remove", &js_sys::Array::new()).await?;
    Ok(true)
}
```

## Verificación

- `cargo check -p vantadb-wasm` ✅ (0 errores, 0 warnings nuevos)
- Sin callers externos de `OpfsFile::delete()` — solo el stub mismo
- `OpfsStorage::delete_file()` sigue siendo la API canónica para delete por path

## Ponytail note

Stub de error reemplazado con implementación real de 2 líneas usando `js_call`. `FileSystemFileHandle.remove()` existe en Chrome 100+ (target de OPFS). No se almacena `dir_handle` en `OpfsFile` — innecesario, el file handle sabe auto-removerse.
