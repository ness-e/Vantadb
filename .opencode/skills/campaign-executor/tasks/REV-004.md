## Contract
`cargo check -p vantadb-openai` and `cargo nextest run --no-run --workspace --build-jobs 2` pass without "tantivy rlib not found" error.

## Atomic Steps
1. Add `"rlib"` to `vantadb-openai/Cargo.toml` `crate-type` — fixes test binary linking by ensuring rlib artifact is produced alongside cdylib

## Skills
ponytail full

## Checks
- `cargo check -p vantadb-openai` passes

## Blast Radius
**CALLERS:** CI workflow, `cargo test`, `cargo nextest`
**CALLEES:** `vantadb` core (transitive through `fjall`, `memmap2`), `pyo3`
**API CHANGES:** None — adding rlib doesn't change the public API or binary output
**TEST IMPACT:** Test binaries can now link against `vantadb_openai` in rlib format

## Investigation Notes
- Root cause: `crate-type = ["cdylib"]` only produces a dynamic library. Test binaries need `rlib` format to link against. When workspace feature unification enables `advanced-tokenizer` (and thus `tantivy`), the missing `rlib` for `vantadb_openai` causes cargo to look for `tantivy` in the wrong format.
- Fix: add `"rlib"` to match pattern used by `vantadb-wasm` (`["cdylib", "lib"]`)
- Error not reproducible on Windows local builds; occurs in Linux CI
